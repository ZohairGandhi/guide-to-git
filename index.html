Guide to Git for the Relatively Clueless

So, you've started studying programming and have finally had enough of pretending to know what's happening when people talk about Git. This might just be the guide for you, young Padawan. This guide will help you set up Git and learn enough of the basics to start feeling a little more in the loop the next time your friends start speaking Greek.

This guide assumes that you are somewhat familiar with using the terminal. If not, you may want to learn some of the basics of using the command line. Trust me, it will open doors beyond just using Git from the terminal. You could even use a terminal editor and one-up your friends.

Git: The Setup

Why did the programmer break up with Git?

Before we begin, a quick distinction: Git is not the same as GitHub. Git is the actual version control system (VCS) that tracks your changes, whereas GitHub is a platform that hosts Git repositories and where Gits come together to enjoy a song and a drink.

Install Git

Head over to the Downloads page on the Git website and follow the instructions to install Git for your operating system. I'll wait here.

Set Up Your Details

Hey, welcome back! Now that you've (hopefully) successfully installed Git, let's set up your user details. These details are included with each commit to track who was responsible for a change so you can blame them when they break the codebase.

If you're a MacOS or Linux user, launch the terminal application. Windows users, launch the Git BASH emulator that should have come packaged with your Git install.

While I have you here Windows users, I must warn you that your future is going to be a little more frustrating than MacOS and Linux users. Developer tooling just seems to work a little less painfully on Unix-based systems. While the Windows experience isn't too bad, if you do get frustrated in the future, you may want to consider dual-booting Linux, using a virtual machine or using the Windows Subsystem for Linux.

Typically, you shouldn't blindly execute terminal commands from the Internet, but you can trust me (Can I borrow a cool $500?). These commands are going to configure the user details for your local Git install. When executing these commands, keep the quotes, but replace the contents inside with your own details. 


git config --global user.name "Peter Parker"
git config --global user.email "peter.parker@dailybugle.com"

Git: The Punchline

Because they couldn't commit!

You've likely worked on an assignment and saved multiple copies of it as assignment_draft.docx, assignment_final.docx and assignment_final_final.docx. With each edit, you probably wanted to save the new version while still keeping the previous ones in case you wanted to come back to them.

This type of versioning can get out of hand when you want to store versions of multiple files. It's also inefficient because each new version contains the contents of the entire file, even the parts that haven't changed since the last version.

This can be resolved using a VCS Ã  la Git. When you save in Git, only the changes since the previous version are recorded and added to a history of all previous versions. Like a time machine, Git lets you go back to previous versions, compare changes and even revert to earlier versions.

Git is a distributed version control system, which is one of three main types of such systems.

Local Version Control

In a local VCS, changes are managed and stored directly on the local machine. If you use Git locally—without hosting your repository on a remote server—it behaves as a local VCS.

A local VCS doesn't really make it easy to share code with collaborators and if your machine fails, you may lose all your data. From experience, that is not enjoyable at all.

Centralised Version Control

These improve upon local systems by featuring a single, central server that hosts all versions of the project files. Individual developers have their own copies of the project but must connect to their server to retrieve the latest version and commit their changes.

This makes collaboration easier and reduces the risk of losing data due to failure of the local machine. But wait...what happens if the server goes down?

Distributed Version Control

We finally arrive at the next iteration of version control. In addition to a central server that hosts the entire project, distributed version control lets each developer clone the entire repository to their local machine—allowing them to work without requiring constant connection with the server.

You can commit changes to your local repository and when ready to push your changes or get others', you can interact with the central server. This makes collaboration possible, eliminates the single point of failure and lets you have a moment to think before you decide if you really want to push that change to the central repository.

If you'd like to read more about version control, I recommend this page on the Git website.

Time to Address the GitHub-Shaped Elephant in the Room

While Git is a standalone piece of software, in practice you will likely be using it in conjunction with GitHub or another platform providing hosting for Git repositories. It just so happens that GitHub is widely used. So, let's finally set up your GitHub profile.

Head over to the GitHub website and create an account. Again, I'll wait.

Back already? Geez, can I have a break? Anyway, now that you've created a GitHub account, you need to link it to your local machine, so you don't have to log in with your credentials each time you want to interact with the remote. This can be done by creating an SSH key pair which consists of two cryptographically linked files.

  • Private key: A secret key that'll reside on your local machine. It's like a digital password only known to the machine.
  • Public key: A key you can share with remote servers like GitHub. It's like a digital lock that will only unlock when the machine attempting to connect provides the corresponding private key.

Run the following command to generate an SSH key. Press Enter through all the prompts. 


ssh-keygen

You can view the generated key using the following command. Leave the terminal open as you will come back to this output in a moment. 


cat ~/.ssh/id_ed25519.pub

Log into your GitHub profile and navigate to the "SSH and GPG keys" section under the Settings. Click the button to create a new SSH key and give it a name that describes the device this key is associated with. Copy the output of your public key from the terminal, paste it into the field on GitHub and click the button to add the key. Congratulations, you now wield a modicum of power in your hands.

Take Git for a Joyride

Let's see what a simple Git workflow looks like. Create a new repository from your GitHub account and name it something descriptive—for example, learning-git. Once the repository is created, click the Code button, then the SSH option and copy the link to your repository.

In your terminal, navigate to the directory where you would like to clone this repository and run the following command. 


git clone your-repo-link

This will create a copy of your repository on your local machine by creating a directory with the same name. Inside this directory, create a file called baby_steps.txt and run git status.

Git will tell you that file is untracked. Now is a good time to introduce the Holy Trinity of Git trees: the HEAD, the Staging Area and the Working Directory. Your file is currently in the Working Directory—the place where files are initially as they are being edited.

When you are ready to commit, run git add baby_steps.txt. This moves the file from the Working Directory to the Staging Area—a kind of waiting room for files before they are sent on their way. Running git status now will tell you that this file is now a change to be committed.

Let's put the therapy to good use and finally commit. Run git commit -m "Take first step towards learning Git" to commit these changes. The -m flag tells Git that this commit will have an associated message which is provided in quotes. There's plenty of advice online about writing good commit messages, but all I'll recommend is to be succinct and meaningful. For example, do not write a message such as "propah progress", which is a commit message I have actually seen.

Upon committing, Git will create a snapshot of the files in the Staging Area and record it in history. HEAD will be updated to this latest commit and the Staging Area and Working Directory will be left clean once again. Your local repository is now one commit ahead of the remote since the changes haven't been pushed yet. Running git log will show you the commit history for this repository along with details such as the time of the commit and the person responsible for this egregiousness.

When you're ready to push your changes to the remote, run git push origin main.

Branch Out a Bit

By default, a repository has a single branch called main and your commits are added onto this branch. However, Git allows you to create other branches and have multiple different versions of your project existing at the same time. Very Christopher Nolan-esque. To create a branch, run git branch branch-name and then git checkout branch-name to switch to this branch so that future commits apply onto it.

To see all the branches in your repository, run git branch. Your current branch is typically shown with an asterisk. Once you've made some commits on a branch and want to apply the changes to main, follow these steps:

  1. Change back to main with git checkout main
  2. Run git merge branch-name to merge the changes from the other branch into main
  3. Trim those extra branches (if not needed) with git branch -d branch-name

If the same file has been changed in different ways on separate branches, you will run into a merge conflict when attempting these steps. In such a case, you'll have to sit back as the branches battle to the death and only the victor remains. If you're not into watching branches fight each other for sport, you can simply resolve the merge conflict yourself. This page from GitHub's documentation does a better job of explaining this than I possibly can, so I'm just linking to it. I know, I'm disappointed in me too.
